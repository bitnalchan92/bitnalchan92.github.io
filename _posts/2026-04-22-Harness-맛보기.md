# 접근 순서

## 전제

- 하네스 엔지니어링은 특정 직군을 위한 기술이 아니라, AI가 적용되는 모든 분야에서 이제 필요한 역량이라고 생각한다.
- 처음부터 너무 많은 스킬을 사용하기보다, 하나의 스킬로 시작해 필요한 부분이 생길 때마다 파이프라인을 고도화해 나가는 것이 올바른 방향이라고 생각한다.



## 거쳐야 할 단계

1. **Clarify** : 사용자의 요구사항을 정확히 '이해'하는 것
   - 이 단계는 반드시 거치는 것이 좋다.
   - 특정 요구사항에 대해 여러 관점에서의 논의점을 제안해달라고 한 뒤, "이게 최선인가?"를 함께 검토하는 과정을 반복하는 것이 중요하다.

2. **Context Gather** : 참고해야 할 관련 문서 / 코드를 찾는 것
   - 요구사항을 명확히 이해한 뒤, 관련된 과거 기록이나 코드를 찾아보게 하는 단계다.
   - 이 단계를 AI에게 통째로 맡기면 정작 참조해야 할 것을 놓치는 경우가 생긴다. 문서에 사용된 단어나 폴더명을 명확하게 정리해두는 것만으로도 이 문제를 상당 부분 해소할 수 있다.

3. **Plan** : 구현 계획 세우기

4. **Generate** : 구현 (코딩)

5. **Evaluate** : 검증



## 궁금한 점

### 1. 왜 task/phase 구조를 사용하나요?

규모가 큰 작업은 한 번의 호흡으로 전부 구현하기 어렵다. 작업을 작게 쪼개 두고, 여러 에이전트 세션이 이어서 작업하도록 구성하기 위해서다.

### 2. 왜 구현에 script를 썼나요?

서브 에이전트에게 작업을 위임하면, 그 오케스트레이션(1번 끝나면 2번 호출, 실패 시 재시도, 중간 상태 저장 등)은 메인 세션의 책임으로 남는다. 이를 Python 스크립트로 분리하면 메인 세션은 스크립트를 실행하기만 하면 되고, 스크립트가 순서대로 에이전트를 호출하고 결과를 기록해준다.

추가적인 장점도 있다. 메인 세션에 오케스트레이션 관련 컨텍스트가 쌓이지 않아 깔끔하게 유지되고, 오케스트레이션 방식을 스크립트로 고정해두면 **모델이 바뀌어도 동일한 흐름으로 작업이 진행된다.**
